# Требования к MVP

## Контекст

Приложение интегрируется с Weeek Public API и использует персональный токен пользователя.
На этапе MVP токен хранится в `localStorage`.
В следующих версиях планируется переход на собственный backend на Go и собственную авторизацию.

Источник по авторизации и формату токена:
- [Weeek Public API](https://developers.weeek.net/)

## Функциональные требования

### FR-01 Экран ввода токена
Система должна отображать стартовую страницу с полем ввода токена Weeek.
Если токен отсутствует, пользователю доступна только эта страница.

### FR-02 Сохранение токена
Система должна сохранять введенный токен в `localStorage` браузера.
После перезагрузки страницы токен должен подставляться автоматически.

### FR-03 Проверка токена через API
Система должна отправлять запросы к Weeek API с заголовком `Authorization: Bearer <token>`.
Если токен невалиден, система должна показать понятную ошибку и запретить доступ к странице задач.

### FR-04 Переход к странице задач
После успешной проверки токена система должна открывать страницу задач на сегодня.

### FR-05 Защита маршрута задач
Если токен отсутствует в `localStorage`, прямой переход на страницу задач должен выполнять редирект на страницу ввода токена.

### FR-06 Получение задач на текущую дату
Система должна запрашивать и отображать задачи, относящиеся к текущей дате (сегодня).

### FR-07 Отображение списка задач
Система должна выводить задачи вертикальным списком.
Минимальный набор данных для карточки задачи: название, дата (или дедлайн), статус (если доступен в API).

### FR-08 Пустое состояние
Если задач на сегодня нет, система должна отображать сообщение: `На сегодня задач нет`.

### FR-09 Состояние загрузки
Во время получения задач система должна отображать индикатор загрузки.

### FR-10 Обработка ошибок API
При сетевой ошибке или ошибке API система должна показывать понятное сообщение и действие для повторного запроса.

### FR-11 Сброс токена
Система должна позволять удалить сохраненный токен (logout/reset), очистить `localStorage` и вернуть пользователя на страницу ввода токена.

### FR-12 Подготовка к будущей авторизации через Go backend
Система должна изолировать работу с токеном в отдельном модуле (например, `auth/tokenStorage`), чтобы позже заменить хранение в `localStorage` на авторизацию через backend без полной переработки UI.

## Out of scope для MVP

- Пользовательская регистрация и логин через собственный backend.
- Ролевая модель и разграничение прав доступа.
- Редактирование, создание и удаление задач в Weeek.
- Фильтры задач по проектам, тэгам, приоритетам и исполнителям.
- Пагинация и бесконечная прокрутка длинных списков.
- Оффлайн-режим и локальное кэширование задач.
- Поддержка нескольких аккаунтов Weeek в одном клиенте.
- Расширенная аналитика и отчеты по задачам.

## Критерии приемки MVP (высокоуровневые)

- Без токена пользователь видит только страницу ввода токена.
- После ввода валидного токена пользователь попадает на страницу задач на сегодня.
- Задачи на сегодня отображаются вертикальным списком.
- При отсутствии задач отображается пустое состояние.
- При ошибках API пользователь получает понятное сообщение и возможность повторить запрос.
